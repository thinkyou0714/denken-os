# 08. 持続性 ＆ KPI(根本原因 #1 ・最大リスク)

## 問題:一人の処理能力を超える前提になっている

毎日: 問題生成 → 検算 → 人の目チェック → 解説作成 → 朝出題 → リプ即返信 →
夜解答 → **さらに自分は二種の勉強** → アプリ開発。

これは現実的に1人の容量を超える。**「全部やろうとして潰れる」のが最大の失敗要因。**
続けられる設計こそ最大のレバレッジ(続けられなければ他の全施策が無意味)。

---

## 対策1:スコープを削る勇気

```
自動化できるもの → 機械に寄せる
  ・問題生成＋検算（アプリの生成エンジン＝ドッグフーディング）
  ・予約投稿（バッチで作って最適時間に自動配信）

人間にしかできないもの → ここに集中
  ・解説の仕上げ・人の目チェック
  ・リプ/会話（アルゴリズムが最も評価する所）
  ・物語を語ること
```
- 「今日の一問」は**毎日でなく週3〜4**で十分(2〜3日に1本でも効果は出る)。
- 出題は半自動、**解説と会話だけ手動**に寄せる。

## 対策2:バッチ運用 SOP

```
週末に1〜2時間:
  □ 1週間分の問題を生成＆検算＆チェックリスト通過(03参照)
  □ 出題・解答テンプレに流し込む(templates/post-templates.md)
  □ 予約投稿をセット(最適時間)
平日:
  □ 投稿は自動。人間はリプ/会話に専念(毎朝30分のリプ営業=05)
常時:
  □ 「検証済み問題」を2週間分ストック → 多忙・体調不良でも途切れない
```

## 対策3:学習＝発信の二毛作

- 自分が勉強した論点を、その日の問題・ログにする。
- 発信用に**別時間を作らない**。学習と発信を一体化する。
- 「なぜこの問題を作ったか」のストーリーも添える(プロセス発信は届きやすい)。

## 対策4:直前期プロトコル / 燃え尽き防止

- 試験直前期は **自分の勉強を最優先**。「ラスト追い込み中につき週○問」と公開
  (失速も「ラストスパート」として共感を呼ぶ)。
- 「疲れたら週1報告だけにする」休止ルールを公開。無理しない姿勢が長期戦の見本。
- 供給をUGC・合格者投稿に分散(`06`)。一人で抱えない。

## KPI(追う指標は3つだけ)

フォロワー数より、Grok時代の「質シグナル」に直結する指標を見る。

| KPI | なぜ | 見る場所 |
|---|---|---|
| **保存数(率)** | いいね以上の最強シグナル。実用価値の証明 | Xアナリティクス |
| **リプ往復数** | いいねの150倍。会話=拡散の核 | 投稿ごとに目視 |
| **フォロー転換** | 物語に共感した人が残ったか | アナリティクス |

補助指標(事業側): コミュニティ参加数、アプリ継続利用率。
> フォロワー数・インプはバニティ指標。手段の自己目的化を防ぐため主指標にしない。

## 週次レビュー テンプレ(コピペ可)

```
【週次レビュー Week ○○】
■ 数字
  ・今週の保存数 TOP3 投稿:
  ・リプが伸びた投稿:
  ・フォロー転換が良かった投稿:
■ 検証(続ける人は「自己否定」でなく「ズレの修正」をする)
  ・当たった型 → 来週も増やす:
  ・外した型 → やめる/直す:
■ 削る判断
  ・今週きつかった作業 → 自動化 or 頻度を落とす:
■ 来週の重点(1つだけ):
```
- 「続かない人は検証せず自己否定に入る」。**検証して淡々とズレを直す**のが継続の本質。

## 四半期レビュー

- 伸びる媒体・型にリソースを集中(固執しない)。
- Xで育った資産を YouTube/note へ展開する比率を数字で調整。
- スコープが膨らんでいないか点検し、**常に削る**。

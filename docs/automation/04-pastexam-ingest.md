# 実装指示 04: 過去問インポート（構造化取込）

> ステータス: 一部実装（核ロジック）。 実装: `lib/ingest/ingest.ts（構造化取込のみ。OCR/PDFパースは別工程）`。README ロードマップ M1 に対応。

## 0. ゴール
公表されている電験の過去問を、出典メタ付きの構造化データ（markdown/JSON）として取り込む。
改題エンジン（01）の素材と、傾向分析（出題頻度）の元データにする。

## 1. 核心の設計判断 ★（著作権＝根本原因対策）
- **出典メタ（年度・期・試験区分）を必須項目**にし、欠落データは取込を弾く。
  `docs/x-strategy/04-compliance.md`：教育目的でも出典明記が利用条件。
- **生の過去問** と **生成/改題問題** を**データ上で明確に分離**（混同が著作権事故の元）。
- 取込時に `source.type=past_exam_quoted` を付与。改題は 01 側で `modified`＋citation。

## 2. スタック
- n8n（README の自動化スタック）または Node スクリプト。
- 数式・回路図対応のため Obsidian markdown（数式/画像）形式で保存（README 方針）。
- OCR/PDF パースが必要なら専用ステップ（数式は手修正前提で flag）。

## 3. 機能要件
- 入力（PDF/HTML）→ 問題単位に分割 → 出典メタ付与 → markdown/JSON 出力
- 数式・回路図はプレースホルダ＋要手修正フラグ（自動OCRの誤りを公開前に止める）
- 重複検出（同一問題の二重取込を防ぐ）
- 公式の「利用上の留意点」へのリンクをメタに記録

## 4. 受け入れ条件
- [ ] 出典メタ欠落データは取込エラーになる
- [ ] 生の過去問と生成問題がストレージ/タグ上で分離されている
- [ ] 数式/図を含む問題に「要手修正」フラグが立つ
- [ ] 重複が検出・スキップされる

## 5. リスク / コンプラ
- 他社問題集（オーム社等）は取込対象外（別著作物）。一次ソース（試験センター公表）のみ。
- OCR 誤りを検算なしで公開しない（01 の検算 or 人手確認を経由）。

## 6. やらないこと
- 他社問題集のスクレイピング
- 数式の完全自動OCR信頼（手修正フラグ必須）
